前言 · 一封給老師的信

親愛的老師：

這 18 週來，我們在黑暗中摸索出的路，原來與 Karpathy 的思考殊途同歸。

讀完 Karpathy 的 gist 時，我感到的不是新奇，而是戰慄。原來這三個月我們不斷爭論的「Domain 為什麼要獨立」、「Log 為什麼要 session 級」，在矽谷最頂尖的 AI 工程師眼裡，正是通往 AGI 知識管理的標準路徑。

這不是一篇筆記，是我們架構的「認證書」。它告訴我們，在 BIM 這個極度碎片化、法律權威至上的領域，我們選擇的路徑是正確的。

一、他說的不是教學：Idea File 的本質

核心主張：Recursive Nature

RAG 是每次 query 重新拼湊碎片；LLM Wiki 是讓 LLM 長期維護一份會累積、會自我交叉引用、會偵測矛盾的 markdown 知識庫。人類只負責策展，LLM 做 bookkeeping。

這份檔案描述的是 Schema，其本身也是 Schema。它告訴 AI 怎麼思考，而非只是怎麼做。

二、三層架構：DNA 的演化邏輯

你不直接編寫 Wiki，你編寫的是「Wiki 怎麼寫」的規則。DNA 對了，Wiki 自己就會長對。

層級角色
Raw sources地基 (不可改)
The Wiki建物 (持續演化)
The Schema設計規範 (共同演化)

三、三種動作：動作基元的無限組合

Ingest 消化、Query 合成、Lint 健檢。所有複雜的行為，都是這三組基本動作的無限遞迴。特別是不 Chunk 的處理方式，讓 AI 具備了完整的上下文推理能力。

四、index 與 log：空間與時序的對話

index.md 是空間性的地圖，在小規模下取代向量庫。log.md 是時序性的日誌，是知識演化的保險。這兩者結合，構成了知識庫的完整座標系。

五、真的，什麼都要變成 Wiki 嗎？

過度的維護是浪費，缺失的維護是負債。Wiki 承載的是「活性知識」：那些會隨著專案演化，但又具備長期參考價值的 BIM 專業邏輯與 SOP。

六、四條判準：決策模型

  1. 權威性：是否涉及法律條文或產業標準？
  2. 頻率：是否為高頻被問到的痛點？
  3. 穩定性：這項共識是否能維持超過一個月？
  4. 跨度：這項知識是否會被多個 Skill 引用？

八、我們刻意不一樣的地方

我們堅持「人類是作者」，因為 BIM 的裁量權不可取代。我們增加了「執行層」，因為知識必須操作 Revit。我們採用「團隊架構」，因為生產力來自於協作而非個人筆記。

九、可以偷回來的四個小東西

三層 Log 防線、/handoff 協議、Nested Metadata 與 AI-agnostic 保底設計。這些不僅是技術，更是制度的具現化。
